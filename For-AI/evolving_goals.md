# Evolving Goals And Decisions

This file is the dated project memory. Add a new dated entry when a chat or implementation changes project direction, product behavior, or release constraints.

## 2026-06-05

- Added `For-AI/` as tracked conceptual memory for future agents.
- The toolkit goal is a public, reusable audio-tactile PPS toolkit centered on Study 5 but extensible to published audio-tactile PPS paradigms.
- The Windows app should feel ready to use, not like a script collection.
- The stimulus/trial designer should separate stimulus design from trial design and save/load settings for reproducibility.
- Randomization should happen in the background. It should not be a primary GUI tool.
- Background randomization should preserve reproducibility, use no-immediate-repeat trial ordering by default, and counterbalance block order for any number of blocks.
- The GUI should become more intuitive: researchers define PPS design parameters, while implementation mechanics stay automatic or documented.
- Literature metadata should only track variables that can be implemented in the GUI, runner, analysis, or template system.
- `Example-configs/Pfeiffer_EJN2018.zip` was added as reference material for Pfeiffer et al. (2018)-style lateral peri-head auditory stimuli. It contains a MATLAB spherical-head/pink-noise simulator and pre-rendered left/right WAVs; future support should distinguish imported baked audio from generated Study 5-style SOFA/HRIR looming.
- Public-release boundaries remain strict: do not commit raw participant recordings, generated participant WAVs, name-bearing outputs, APKs, VR code, model downloads, SOFA/HRIR files, background music, or third-party assets without verified redistribution rights.
- The 3DTI render contract now records explicit listener head geometry and gain assumptions. Render configs/manifests carry head diameter/radius, 3DTI high-quality anechoic model details, customized ITD, propagation delay, distance attenuation, near-field effect, and relative digital level normalization. The Pfeiffer preload aligns 3DTI listener head radius with the reference simulator's `head_diam = 0.18 m`; Pfeiffer dB constants are stored as provenance only because the reference script peak-normalizes its WAV output.
- The Qt designer now has three workflow tabs: `Stimulus Design`, `Trial Assembler`, and `Experiment Runner`. `Trial Assembler` replaces the flat trial-design view with condition controls, trial-family toggles, block assembly, trial preview, and participant block-order preview. The runner tab started as a preflight/launcher, but the current target is an integrated Qt Prepare/Run/Review workflow while keeping `pps-run` as a compatibility path.
- The published-paradigm preload library was initially stress-tested and expanded to 17 profiles. Partial profiles covered Ferri 2015, Taffou 2014, Teneggi 2013, Serino 2015 peri-trunk/peri-hand/front-back variants, Galli 2015 wheelchair PPS, Noel 2015 walking PPS, and Lerner 2021 3D PPS. Profiles marked `partial` are structured runnable scaffolds only; exact replication still requires missing GUI/model support such as proprietary ecological sounds, social/locomotor/VR factors, speaker-array crossfades, body-scaled distances, and multi-direction batch trajectories.

## 2026-06-06

- The published-paradigm preload stress test now contains 20 profiles. New partial profiles cover Canzoneri 2013 tool-use reshaping, Canzoneri 2013 amputation/prosthesis PPS, and Serino/Canzoneri 2015 synchronous versus asynchronous far-audio/hand-tactile training.
- `canzoneri_2012_dynamic_sounds` was corrected toward the original accessible method values: 3000 ms pink noise with 1000 ms pre/post silence, T1-T5 at 300/800/1500/2200/2700 ms, tactile-only T0/T6 baseline metadata, 8 target repetitions per timing, and 76 auditory-only catch trials. Exact replication is still blocked by the current scheduler lacking direction-coupled T0/T6 baseline subtypes.
- Added `docs/PUBLISHED_PARADIGM_STRESS_TEST.md` as the current per-study audit for trial families, SOA/spatial semantics, jitter/procedure gaps, and missing implementation parameters.
- Keep the GUI profile-driven. Do not expose every literature parameter as a top-level option. Future schema work should group rare replication details into `TrialFamilySpec`, `TimingPolicySpec`, `BlockFactorSpec`, `PhaseSpec`, `TactileStimulusSpec`, `DistanceUnitSpec`, and `AnalysisSpec`, with a read-only Profile Details checklist plus collapsed advanced overrides.
- Static near/far audio-tactile APPS studies and sound-only looming motor-preparation studies remain deferred candidates until the app supports static spatial trial families or a separate auditory-only profile category.
- Runner audio routing now distinguishes legacy Study 5 stereo blocks from rendered binaural+tactile blocks. Legacy stereo files remain WAV-left tactile/WAV-right audio and are routed to physical output 2/1. Rendered trajectory WAVs are channel 0 left ear, channel 1 right ear, channel 2 tactile, and require one synchronized 3+ channel output device.
- On the tested Komplete Audio 6 Windows setup, `SD_ENABLE_ASIO=1` exposes `Komplete Audio ASIO Driver` as a 6-output ASIO endpoint; the separate Windows `Output 1/2` and `Output 3/4` endpoints are stereo-only and are legacy-only for this project. Do not split rendered binaural+tactile playback over separate stereo endpoints.
- Added `pps-audio-stress` / `windows\Stress_Audio_Device.bat` to run silent device-open/write tests across channel counts, latencies, and block sizes. The local focused stress matrix for Komplete ASIO device 31 passed 3-, 4-, and 2-channel silent playback for all tested combinations and recommends 3-channel ASIO output with physical outputs 1/2 for headphones and output 3 for tactile.
- WASAPI loopback recording may not capture ASIO multichannel playback. For full rendered-stimulus QC, use hardware loopback from physical outputs or treat the stress matrix plus render manifest as device-open validation.
- Driver audit found the installed Komplete Audio 6 MK2 driver is already NI `5.22.0.17558`, matching the latest official NI listing (`Komplete Audio 6 MK2 Driver 5.22.0 - Windows 10`). NI's direct CDN ZIP downloads returned HTTP 403 to scripted requests, so reinstall/update should use the official browser page or Native Access. ASIO4ALL was tested and is stereo-only here; Nahimic WDM-KS exposes virtual multichannel output but is not the physical lab routing path.
- The experiment player now uses a single persistent Komplete ASIO output stream for block playback, spoken instructions, background audio, and click/tactile feedback. This is required because the ASIO device fails when a persistent click stream and a separate block stream are opened concurrently. Selected stable runtime defaults are 3 channels, requested latency `0.010`, and blocksize `256`; a 60-second callback stress passed at about 16.5 ms actual latency, while `0.003`/64 failed longer persistent-stream testing despite passing short sweeps.
- The Qt `Experiment Runner` tab is no longer a launcher-only page. It now exposes a compact Prepare/Run/Review workflow: participant ID, rendered-stimulus status, fixed audio-route status, session-output path, `Render`, `Prepare`, `Stress Audio`, and `Start Focus Mode`. It writes run packages under `local_data\sessions\<participant_id>_<timestamp>\` containing `design.json`, `protocol_schedule.csv`, `session_manifest.json`, per-block manifest CSVs/WAVs, `events.csv`, `events.xdf`, `analysis_summary.txt`, and immediate analysis CSVs. `pps-run` remains as the legacy Study 5 compatibility runner.
- The current Qt session package materializes block WAVs by concatenating the rendered looming WAV selected for each protocol row. This gives the GUI a real end-to-end run path, but the renderer still needs a future per-trial tactile/SOA render expansion if exact multi-SOA block audio is required from a single design without relying on the current rendered cue track.
- Focus Mode timing policy changed: direct event timing is primary, audio decoding is backup. The `TimingEventHub` writes actual events to the local CSV/XDF logger and optionally to an LSL `PPSMarkers` outlet. Planned tactile events are anchored to `audio_sample_zero` emitted by the audio callback, not to the earlier playback request. Mouse clicks are logged immediately as `mouse_click`; the tactile-channel response marker is a low-gain physical QC pulse only, intended for hardware loopback recovery if the event log fails.
- Windows setup now installs the optional `lsl` extra. If `pylsl` or LabRecorder is unavailable, sessions still write internal `events.xdf`; manifests/logs should record LSL availability rather than failing the experiment.
- The Qt designer now uses nested resizable split panels across `Stimulus Design`, `Trial Assembler`, and `Experiment Runner`. Splitter sizes persist locally through Qt settings, each tab has `Reset Layout`, and the stimulus viewer has `Maximize Preview` / `Restore` for large 2D/3D trajectory inspection. This remains the Qt comparison/fallback path after the local HTML dashboard became the standard researcher UI.
- The Qt designer now uses functional color-coded panes inspired by experiment-builder interfaces. Colors distinguish trajectory start/end, timing, viewer, trial conditions/families/blocks, schedule previews, and runner panes; they remain UI-only and are not experiment parameters. A later aesthetic pass moved the app away from native Windows chrome by forcing Qt Fusion style, modernizing inputs/tables/buttons/splitter handles, compacting dense endpoint/timing/condition controls, and preserving resizable workspaces with versioned layout defaults. Study-profile labels now foreground paper-style citation information, and the profile bar includes a `Citation` menu to show the source citation or export BibTeX/CSL JSON generated from template metadata.
- The app now has a packaged PPS Toolkit visual identity: SVG source plus generated PNG/ICO assets under the Python package. The Qt designer and native Focus Mode set this icon at runtime, the Windows shortcut points to the packaged `.ico`, the local dashboard uses the same mark as its favicon, and the designer batch launcher prefers `pythonw.exe` to avoid showing a separate generic Python console window.

Treat these manuals/spec pages as third-party material: do not move them into tracked release assets unless redistribution rights are reviewed. The repo identifies the lab audio interface as Komplete Audio 6 MK2; the exact Woojer Strap generation is still unresolved, so cached Woojer references include Strap 3 plus current Strap 4/family docs. ## 2026-06-08

- Dashboard render/bake reliability was stress-tested for the GUI-to-3DTI handoff. The browser payload now reuses the same normalized trajectory controls used by the live preview, so Apply, Bake Stimulus, Render, and legacy spatial fallback values no longer diverge from preview for clamped distances, normalized rotations, or bounded movement/hold durations.
- Added dashboard stress coverage for static JS payload contracts, repeated API-level GUI trajectory/source/SOA handoff into `render_config.3dti.json`, render and bake endpoint design snapshots, and one native 3DTI measured WAV check for a lateral trajectory. The native check verifies 3 output channels, expected frame count, tactile onset at the requested SOA, non-clipping audio, manifest hashes, and left/right dominance reversal.
- Current evidence supports the software handoff from dashboard decisions to saved design, 3DTI-compatible config, native 3DTI wrapper output, and silent Komplete ASIO 3-channel device opening. It does not prove physical headphone/tactile acoustics without electrical loopback capture on the lab wiring; use `pps-latency-validate` for that measurement boundary.
- The HTML dashboard now treats baseline strategy as a separate forced decision segment before Trial Designer and before Run Setup. The supported visible tactics are no baseline, tactile-only timing anchors, SOA 0, sound-offset/end anchor, and custom baseline timings. The segment exposes default catch/baseline proportions and estimates block composition/duration; each Trial Designer row then carries explicit `audio_tactile_percentage`, `catch_percentage`, and `baseline_percentage` fields. The backend scheduler uses those row percentages to materialize catch and baseline rows per trial type row, while legacy designs without row percentages continue using the older global catch/baseline path. Protocol CSVs and prepared block manifests now include the row composition percentages.
- Source-level stimulus trajectory inventory is now part of the HTML dashboard contract. `NoiseDefinition` and `AudioFileSpec` can carry `trajectory_snapshot` metadata; the dashboard companion fills missing snapshots for all preload profiles from the profile trajectory at load time and records the exact current GUI trajectory when a new source is baked. The embedded Three.js trajectory preview, not the HTML source cards, owns the 2D/3D visual representation of baked and preloaded source trajectories. Stimulus source cards should carry compact metadata, local file actions, and hidden trajectory snapshots for payload round-tripping; the viewer draws colored overlay paths, including parallel multi-color traces when different tones share the same trajectory.
- The preload library is now a bundled local catalog rather than a Study-5-only asset set. `tools/build_preload_catalog.py` materializes every preload profile into the segment-mirrored folder structure, writes per-segment metadata files, prebakes auditory-only profile WAVs into `02_looming_stimuli/`, and rebuilds the inventory with hashes, tone metadata, trajectory snapshots, and catalog segment paths. Procedural source definitions can carry `prebaked_path`, imported/custom looming files can carry `tone_type`, and source cards use multi-color trajectory traces when multiple tones share one trajectory.
- Direction factors in preload profiles must not be collapsed to a single source path when the geometry can be represented. The preload catalog builder now expands reversible/mirrored direction labels such as left-to-right/right-to-left, looming/receding, front-to-back/back-to-front, and rear-left/rear-right into separate auditory-only WAVs and trajectory snapshots. The Pfeiffer EJN2018 lateral profile now exposes both left-to-right and right-to-left variants in the viewer.
- Lerner 2021 is a twelve-direction 3D boundary profile, not a two-source profile. Its preload now materializes the twelve Table-1 virtual source directions on the R = 1.2 m sphere for both dynamic pink and flat white source types, yielding 24 local auditory-only source assets and 12 unique viewer geometries. When the local companion loads direction-specific catalog assets, it collapses `protocol.auditory_motion_directions` to `source_trajectory` so schedules do not double-cross baked source direction with a second direction factor.
- Published-study DOI display now includes a separate replication caveat below the DOI URL. The DOI box remains DOI-only, but published profiles warn that the dashboard uses local toolkit recreations of the study's reported parameters and bundled/generated assets, not the exact original stimuli used in those studies. Study 5/custom profiles do not show this published-study caveat.
- Baked generated and imported/local stimulus source cards are now a wrapped, side-by-side inventory grid rather than a vertical file list. Cards are color-coded by noise/tone type; generated noise cards inherit their procedural color and imported/custom audio cards expose a persisted `tone_type`/box-color selector. Local file paths stay hidden as backend payload metadata; researchers inspect files through an `Open Folder` action handled by the local companion.
- The baseline strategy segment should stay compact. It now asks only whether baseline trials are used and which baseline tactic applies, using checkbox/card controls for matched SOA anchors, sound-onset/min-SOA anchors, sound-offset/max-SOA anchors, and custom timing anchors. The segment keeps only a small impact strip for timing anchor, baseline trials per block, total trials per block, and estimated minutes; detailed row composition remains in Trial Designer.
- Trial Designer is moving from a generic filmstrip editor to a block-based row builder. In custom mode an empty Trial Rows panel should show a large plus action. Each row is a trial type assembled left-to-right from fixed blocks and randomizer blocks. Fixed blocks choose one saved custom clip and always play it at that position. Randomizer blocks choose from previously created stimulus sources, keep SOA values inside the block, and display vertical source rows plus baseline/catch rows with numeric per-block counts. The randomizer is the visible place where selected source x SOA combinations plus catch/baseline rows become the counterbalanced set. For now the backend supports one randomized stimulus slot per row, so the UI should prevent multiple randomizer blocks in the same row while still allowing additional fixed blocks. Study 5 uses two sequential rows: `Inhale instruction` fixed block plus all four frontal looming sources, then `Exhale instruction` fixed block plus the same looming-source randomizer, meaning the prepared block alternates/cycles inhale-preceded and exhale-preceded trial types.

## 2026-06-09

- The HTML dashboard workflow now separates trial sequence decisions from trial amount decisions. Segment 3 is Baseline Strategy only: the user picks exactly one established tactic through mutually exclusive checkbox/cards, with no visible baseline percentages, timing-value editor, trial counts, or duration analytics. Segment 4 is Trial Sequence Design only: users assemble the within-trial row sequence and randomization source set with minimalist plus affordances and small `x` removals. It must not show repetitions, block count, SOA totals, per-source trial counters, baseline counters, catch counters, or trial-total summaries. Segment 5 is Trial-Block Design: SOA values, repetitions, block count, trial preview, block-level counterbalancing, catch/baseline/audio-tactile amounts, and duration/count summaries. The custom workflow gate now has a separate `block` step so a complete sequence without SOA/repetition/block decisions stops at Segment 5 before render/prepare.
