# Evolving Goals And Decisions

This file is the dated project memory. Add a new dated entry when a chat or implementation changes project direction, product behavior, or release constraints.

## 2026-06-05

- Added `For-AI/` as tracked conceptual memory for future agents.
- The toolkit goal is a public, reusable audio-tactile PPS toolkit centered on Study 5 but extensible to published audio-tactile PPS paradigms.
- The Windows app should feel ready to use, not like a script collection.
- The stimulus/trial designer should separate stimulus design from trial design and save/load settings for reproducibility.
- Randomization should happen in the background. It should not be a primary GUI tool.
- Background randomization should preserve reproducibility, use no-immediate-repeat trial ordering by default, and counterbalance block order for any number of blocks.
- The GUI should become more intuitive: researchers define PPS design parameters, while implementation mechanics stay automatic or documented.
- Literature metadata should only track variables that can be implemented in the GUI, runner, analysis, or template system.
- `Example-configs/Pfeiffer_EJN2018.zip` was added as reference material for Pfeiffer et al. (2018)-style lateral peri-head auditory stimuli. It contains a MATLAB spherical-head/pink-noise simulator and pre-rendered left/right WAVs; future support should distinguish imported baked audio from generated Study 5-style SOFA/HRIR looming.
- Public-release boundaries remain strict: do not commit raw participant recordings, generated participant WAVs, name-bearing outputs, APKs, VR code, model downloads, SOFA/HRIR files, background music, or third-party assets without verified redistribution rights.
- The 3DTI render contract now records explicit listener head geometry and gain assumptions. Render configs/manifests carry head diameter/radius, 3DTI high-quality anechoic model details, customized ITD, propagation delay, distance attenuation, near-field effect, and relative digital level normalization. The Pfeiffer preload aligns 3DTI listener head radius with the reference simulator's `head_diam = 0.18 m`; Pfeiffer dB constants are stored as provenance only because the reference script peak-normalizes its WAV output.
- The Qt designer now has three workflow tabs: `Stimulus Design`, `Trial Assembler`, and `Experiment Runner`. `Trial Assembler` replaces the flat trial-design view with condition controls, trial-family toggles, block assembly, trial preview, and participant block-order preview. The runner tab started as a preflight/launcher, but the current target is an integrated Qt Prepare/Run/Review workflow while keeping `pps-run` as a compatibility path.
- The published-paradigm preload library was initially stress-tested and expanded to 17 profiles. Partial profiles covered Ferri 2015, Taffou 2014, Teneggi 2013, Serino 2015 peri-trunk/peri-hand/front-back variants, Galli 2015 wheelchair PPS, Noel 2015 walking PPS, and Lerner 2021 3D PPS. Profiles marked `partial` are structured runnable scaffolds only; exact replication still requires missing GUI/model support such as proprietary ecological sounds, social/locomotor/VR factors, speaker-array crossfades, body-scaled distances, and multi-direction batch trajectories.

## 2026-06-06

- The published-paradigm preload stress test now contains 20 profiles. New partial profiles cover Canzoneri 2013 tool-use reshaping, Canzoneri 2013 amputation/prosthesis PPS, and Serino/Canzoneri 2015 synchronous versus asynchronous far-audio/hand-tactile training.
- `canzoneri_2012_dynamic_sounds` was corrected toward the original accessible method values: 3000 ms pink noise with 1000 ms pre/post silence, T1-T5 at 300/800/1500/2200/2700 ms, tactile-only T0/T6 baseline metadata, 8 target repetitions per timing, and 76 auditory-only catch trials. Exact replication is still blocked by the current scheduler lacking direction-coupled T0/T6 baseline subtypes.
- Added `docs/PUBLISHED_PARADIGM_STRESS_TEST.md` as the current per-study audit for trial families, SOA/spatial semantics, jitter/procedure gaps, and missing implementation parameters.
- Keep the GUI profile-driven. Do not expose every literature parameter as a top-level option. Future schema work should group rare replication details into `TrialFamilySpec`, `TimingPolicySpec`, `BlockFactorSpec`, `PhaseSpec`, `TactileStimulusSpec`, `DistanceUnitSpec`, and `AnalysisSpec`, with a read-only Profile Details checklist plus collapsed advanced overrides.
- Static near/far audio-tactile APPS studies and sound-only looming motor-preparation studies remain deferred candidates until the app supports static spatial trial families or a separate auditory-only profile category.
- Runner audio routing now distinguishes legacy Study 5 stereo blocks from rendered binaural+tactile blocks. Legacy stereo files remain WAV-left tactile/WAV-right audio and are routed to physical output 2/1. Rendered trajectory WAVs are channel 0 left ear, channel 1 right ear, channel 2 tactile, and require one synchronized 3+ channel output device.
- On the tested Komplete Audio 6 Windows setup, `SD_ENABLE_ASIO=1` exposes `Komplete Audio ASIO Driver` as a 6-output ASIO endpoint; the separate Windows `Output 1/2` and `Output 3/4` endpoints are stereo-only and are legacy-only for this project. Do not split rendered binaural+tactile playback over separate stereo endpoints.
- Added `pps-audio-stress` / `windows\Stress_Audio_Device.bat` to run silent device-open/write tests across channel counts, latencies, and block sizes. The local focused stress matrix for Komplete ASIO device 31 passed 3-, 4-, and 2-channel silent playback for all tested combinations and recommends 3-channel ASIO output with physical outputs 1/2 for headphones and output 3 for tactile.
- WASAPI loopback recording may not capture ASIO multichannel playback. For full rendered-stimulus QC, use hardware loopback from physical outputs or treat the stress matrix plus render manifest as device-open validation.
- Driver audit found the installed Komplete Audio 6 MK2 driver is already NI `5.22.0.17558`, matching the latest official NI listing (`Komplete Audio 6 MK2 Driver 5.22.0 - Windows 10`). NI's direct CDN ZIP downloads returned HTTP 403 to scripted requests, so reinstall/update should use the official browser page or Native Access. ASIO4ALL was tested and is stereo-only here; Nahimic WDM-KS exposes virtual multichannel output but is not the physical lab routing path.
- The experiment player now uses a single persistent Komplete ASIO output stream for block playback, spoken instructions, background audio, and click/tactile feedback. This is required because the ASIO device fails when a persistent click stream and a separate block stream are opened concurrently. Selected stable runtime defaults are 3 channels, requested latency `0.010`, and blocksize `256`; a 60-second callback stress passed at about 16.5 ms actual latency, while `0.003`/64 failed longer persistent-stream testing despite passing short sweeps.
- The Qt `Experiment Runner` tab is no longer a launcher-only page. It now exposes a compact Prepare/Run/Review workflow: participant ID, rendered-stimulus status, fixed audio-route status, session-output path, `Render`, `Prepare`, `Stress Audio`, and `Start Focus Mode`. It writes run packages under `local_data\sessions\<participant_id>_<timestamp>\` containing `design.json`, `protocol_schedule.csv`, `session_manifest.json`, per-block manifest CSVs/WAVs, `events.csv`, `events.xdf`, `analysis_summary.txt`, and immediate analysis CSVs. `pps-run` remains as the legacy Study 5 compatibility runner.
- The current Qt session package materializes block WAVs by concatenating the rendered looming WAV selected for each protocol row. This gives the GUI a real end-to-end run path, but the renderer still needs a future per-trial tactile/SOA render expansion if exact multi-SOA block audio is required from a single design without relying on the current rendered cue track.
- Focus Mode timing policy changed: direct event timing is primary, audio decoding is backup. The `TimingEventHub` writes actual events to the local CSV/XDF logger and optionally to an LSL `PPSMarkers` outlet. Planned tactile events are anchored to `audio_sample_zero` emitted by the audio callback, not to the earlier playback request. Mouse clicks are logged immediately as `mouse_click`; the tactile-channel response marker is a low-gain physical QC pulse only, intended for hardware loopback recovery if the event log fails.
- Windows setup now installs the optional `lsl` extra. If `pylsl` or LabRecorder is unavailable, sessions still write internal `events.xdf`; manifests/logs should record LSL availability rather than failing the experiment.
- The Qt designer now uses nested resizable split panels across `Stimulus Design`, `Trial Assembler`, and `Experiment Runner`. Splitter sizes persist locally through Qt settings, each tab has `Reset Layout`, and the stimulus viewer has `Maximize Preview` / `Restore` for large 2D/3D trajectory inspection. This remains the Qt comparison/fallback path after the local HTML dashboard became the standard researcher UI.
- The Qt designer now uses functional color-coded panes inspired by experiment-builder interfaces. Colors distinguish trajectory start/end, timing, viewer, trial conditions/families/blocks, schedule previews, and runner panes; they remain UI-only and are not experiment parameters. A later aesthetic pass moved the app away from native Windows chrome by forcing Qt Fusion style, modernizing inputs/tables/buttons/splitter handles, compacting dense endpoint/timing/condition controls, and preserving resizable workspaces with versioned layout defaults. Study-profile labels now foreground paper-style citation information, and the profile bar includes a `Citation` menu to show the source citation or export BibTeX/CSL JSON generated from template metadata.
- The app now has a packaged PPS Toolkit visual identity: SVG source plus generated PNG/ICO assets under the Python package. The Qt designer and native Focus Mode set this icon at runtime, the Windows shortcut points to the packaged `.ico`, the local dashboard uses the same mark as its favicon, and the designer batch launcher prefers `pythonw.exe` to avoid showing a separate generic Python console window.
- Hardware vendor docs for Komplete Audio 6 and Woojer Strap are now indexed in `docs/HARDWARE_VENDOR_DOCS.md`, with local downloaded snapshots in an ignored hardware-doc cache plus a manifest of source URLs and SHA-256 hashes. Treat these manuals/spec pages as third-party material: do not move them into tracked release assets unless redistribution rights are reviewed. The repo identifies the lab audio interface as Komplete Audio 6 MK2; the exact Woojer Strap generation is still unresolved, so cached Woojer references include Strap 3 plus current Strap 4/family docs. Public docs should avoid literal private-path markers because release audit treats them as forbidden publication-boundary strings.
- Added a local HTML dashboard as the standard researcher-facing UI while keeping the Qt designer available for comparison and fallback. Launch with `pps-dashboard` or `windows\Launch_HTML_Dashboard.bat`; it serves a local-only browser UI from `127.0.0.1` through a small FastAPI backend and opens the user's default browser automatically. The dashboard launcher is idempotent: if the local app is already running on the requested port, it opens the existing URL instead of starting a duplicate server. The dashboard covers researcher-facing design decisions, trajectory preview, noise/custom-audio setup, trial previews, render, prepare, audio stress, native Focus Mode launch, and review. Timing-sensitive participant running remains native/Python-backed via `pps-focus --session-manifest ...`, not browser JavaScript. Windows setup now installs the `web` extra so both UIs can be compared after the normal setup.
- HTML dashboard feedback: the box under the Study Profile preload selector should contain only the clickable full DOI URL (`https://doi.org/...`) for the selected published profile. Do not include citation text, loaded/pending status, notes, or source fallbacks in that box; hide it when no DOI is available.
- HTML dashboard feedback: both the Stimulus Design panel and the Trajectory Preview panel should use compact 2D/3D segmented buttons for view switching; do not use a dropdown for this mode control.
- HTML dashboard feedback: stimulus trajectory fields and the embedded trajectory preview should stay live-synchronized without requiring Apply Design. Editing start/end distance, start/end rotation, or movement duration updates the 2D/3D preview immediately. In 2D mode, the green start marker and red end marker are both draggable with the mouse; dragging either marker updates the matching distance and rotation fields in the Stimulus Design panel. Saving still happens through Apply/Continue/Render rather than silent autosave while typing.
- HTML dashboard trajectory preview behavior: in 2D mode, the camera/view scale should stay stable while the user edits trajectory values. Radius and endpoint changes update the geometry but should not automatically resize the view. The 2D view is a fixed top-down editing surface with draggable start/end points, bounded background panning, explicit `+`/`-` zoom buttons, and a `Fit Radius` action that recenters and scales the whole radius circle into view. `Reset Camera` may also return to this fitted centered view.
- HTML dashboard feedback: the Study Profile dropdown is the load action. Do not show a separate Load Profile button; selecting a published preload or Custom design (define manually) immediately loads that profile through the backend.
- HTML dashboard feedback: the Study Profile preload selector should include `Custom design (define manually)` as a first-class option. Loading it clears published-profile metadata, starts a generic manual design, and reminds the researcher that trajectory, timing, audio, trial assembly, and session settings must be defined manually.
- Windows end-user launch direction: `windows\Create_Desktop_Shortcut.ps1` now targets `windows\Launch_HTML_Dashboard.bat`; `windows\Launch_PPS_App.bat` remains the legacy locked Study 5 runner rather than the default researcher dashboard.
- HTML dashboard workflow direction: the left rail should stay fixed/floating as one-page navigation with smooth scroll behavior, but it should not contain panel-size sliders. Panel sizing should happen directly in the workspace through draggable, snapping splitters/gutters and panel-edge handles, closer to a Blender-style adjustable workspace. Custom mode now starts deliberately incomplete and enforces a sequential setup gate: Study Profile, Stimulus Design, Trial Assembly, Run Preparation, then Review. Backend render/prepare endpoints reject incomplete custom profiles.
- New website direction: the HTML dashboard can be published through GitHub Pages as a static interface. Hosted pages should use relative dashboard/viewer asset paths and talk to a local PPS companion backend on `127.0.0.1:8766`; they must not attempt to install packages or run timing-sensitive experiments directly in browser JavaScript. `windows\Start_Website_Companion.bat` starts the local backend without opening the local UI, and the dashboard left rail includes companion connection controls plus a software download link.
- Added `pps-latency-validate` as the CLI-only electrical loopback validation path for the Komplete Audio 6 MK2 ASIO route. The protocol assumes Woojer Strap 4 as the wired analog tactile target, documents calibration wiring as physical output 1/2/3 to input 1/2/3, writes validation artifacts under `artifacts\latency_validation\`, stores local baselines under `local_data\latency_baselines\`, and explicitly does not claim mechanical Woojer vibration latency.
- New maintenance protocol: every change to the HTML/dashboard web GUI must also update the matching online website/static GitHub Pages version before the work is considered complete. Agents should verify the hosted/static version still uses relative dashboard/viewer assets and communicates with the local companion backend rather than moving timing-sensitive experiment execution into browser JavaScript.
- HTML/dashboard local-orchestration rule: the browser UI, including the GitHub Pages version, must never upload local stimulus files, participant data, generated WAVs, or experiment artifacts online. File imports are local companion/backend actions on the research PC and should store ignored local copies for the design/render/session pipeline. The Stimulus Design panel exposes a Stimulus Selection module where procedural white/pink/blue/violet/brown noise and imported dry/control audio are staged, then baked by the local backend before they become editable/usable stimulus source boxes. Instruction snippets are imported locally as fixed clips for trial assembly, not as looming sources.
- HTML dashboard stimulus-selection direction: the separate Custom Stimulus Builder / sequence panel was removed because it made the Stimulus Design workflow too complicated. Researchers stage generated noise, dry custom tones, or already-looming/control audio through Stimulus Selection, then use the local Bake Stimulus action before those items become usable looming sources. Instruction snippets are imported through the same area as fixed clips and still carry placement (`before`/`after`), target source, respiratory phase, and gap metadata. Ordering, timing-sensitive playback, and any final audio assembly remain backend/native responsibilities rather than drag/drop browser controls.
- HTML dashboard stimulus-baking rule: generated noise or imported dry/control audio should not become a trial-assembly source merely because it was selected. The GUI should stage a source recipe, call the local backend to bake the looming auditory stimulus through the selected trajectory/3DTI-compatible render path, and only add the source after the bake succeeds. Baked source WAVs should be auditory-only/binaural; tactile cue channels are not present at this stage and should be added later from the trial/SOA schedule during pre-run session/block preparation. Focus Mode should consume prepared WAVs/manifests rather than generating stimuli during timed participant execution.
- HTML dashboard baked-source behavior: source cards for baked/rendered stimuli should expose an `Open Folder` action so researchers can inspect the local WAV location. This must be a local companion backend action such as `/api/local/open-folder`, not a direct hosted-page upload or browser-owned experiment action.
- HTML dashboard layout direction: keep the one-page website behavior, but group controls into sequential decision segments. Segment 1 contains only the study preload/custom-profile decision and design naming. Segment 2 contains the looming-stimulus builder, including trajectory numeric fields, 2D/3D preview, source staging/baking, baked-source cards, and backend feedback for bake/render progress. Segment 3 contains the trial designer, including filmstrip rows, SOAs, repetitions, blocks, catch percentage, and trial preview. Participant count, participant ID, participant order preview, prepare/stress/focus actions, jobs, and review belong at the bottom in the run/session area.
- Study 5 is the user's unpublished local lab profile, not the Canzoneri published profile. The preload template `study5_box_breathing_pps` carries the bundled owned 4.000 s inhale/exhale instruction WAVs from `assets/breathing/`, Study 5 trajectory/SOA defaults, and two filmstrip rows: `Inhale instruction | Looming Stimulus` and `Exhale instruction | Looming Stimulus`. Canzoneri remains a separate published-study scaffold. The HTML dashboard should treat Study 5 as the default preload moving forward: it appears first in the preload list and fresh or scratch-custom startups initialize from this profile rather than the generic/default design.
- Preload assets are now managed through `assets/preloads/preload_inventory.json`. Study 5 ships with owned, bundled, auditory-only 4.000 s looming WAVs for pink, blue, white, and brown frontal sources under `assets/preloads/study5_box_breathing_pps/`; its template uses these as preserved local looming files rather than procedural noise recipes. The profile-specific manifest records the original generated-noise recipe, trajectory, hashes, and the rule that tactile is deferred until session preparation. Other tracked preload profiles are indexed as recipe-only until their assets are approved for bundling or baked by the local companion. The browser can show/request asset status, but verification, download, and baking remain local companion backend actions.
- Added `For-AI/skills/html-dashboard-orchestrator/` as a project-local, Codex-skill-shaped workflow for future HTML dashboard changes. Use it to keep browser GUI edits connected to backend contracts, local-only file handling, GitHub Pages synchronization, browser validation, and concise project-memory updates.
